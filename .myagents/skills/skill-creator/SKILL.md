# Skill Creator

创建技能并迭代改进的工作流。

## 核心循环

```
理解意图 → 起草技能 → 运行测试 → 用户评审 → 改进
```

每一步都可以灵活跳入跳出，用户说"不需要测试，直接来"就跳过测试部分。

---

## 创建技能

### 1. 理解意图

通过对话提取：
1. 这个技能要解决什么问题？
2. 什么时候触发？（用户会怎么说？）
3. 期望的输出格式是什么？
4. 需要测试用例验证效果吗？

**主动提问**：边缘情况、输入输出格式、成功标准、依赖工具。不要急着写，等信息清晰。

### 2. 起草 SKILL.md

根据对话填充以下组件：

- **name**：技能标识符，简洁
- **description**：这是主要触发机制——必须同时包含"做什么"和"何时触发"。注意 Claude 有"触发不足"倾向，描述可以稍微"pushy"一点
- **body**：Markdown 指令

#### 技能结构

```
skill-name/
├── SKILL.md (必须)
│   ├── YAML frontmatter (name, description 必须)
│   └── Markdown instructions
└── Bundled Resources (可选)
    ├── scripts/    - 确定性/重复性任务的脚本
    ├── references/ - 按需加载的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露原则

| 层级 | 内容 | 理想行数 |
|------|------|----------|
| 元数据 | name + description | ~100 词 |
| SKILL.md body | 触发时加载 | <500 行 |
| Bundled resources | 按需加载 | 不限 |

**关键原则：**
- SKILL.md 控制在 500 行以内；接近时应分层并给出清晰指引
- 超过 300 行的引用文件加目录表
- 引用文件在 SKILL.md 中明确说明何时读取

#### 技能描述写作

❌ 模糊：`"当需要处理文档时使用"`

✅ 高信号：`"用户提到 Word doc、.docx、或要求生成报告"`

**描述要点：**
- 包含"做什么" + "何时触发"
- 具体到输入形式（GitHub 链接、PDF 文件、URL...）
- 具体到输出形式（报告、文档、代码...）
- 可以稍微"pushy"，触发不足比过度触发更常见

### 3. 测试用例

写完草稿后，设计 2-3 个真实测试场景（用户会实际说的话）。与用户确认后再运行。

保存到 `evals/evals.json`：

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务描述",
      "expected_output": "期望结果描述",
      "files": []
    }
  ]
}
```

## 运行和评测

### 执行循环（不跳过）

```
起草技能 → 运行测试 → 获取反馈 → 改进 → 重复
```

### 测试目录结构

在技能目录同级的 `<skill-name>-workspace/` 下组织结果：

```
iteration-1/
├── eval-0/
│   ├── with_skill/outputs/
│   └── eval_metadata.json
└── eval-1/
    └── ...
```

### 评测步骤

1. **运行测试** — 每次测试spawn with-skill 和 baseline 两个子agent，并行运行
2. **获取反馈** — 用 eval-viewer 或直接展示结果给用户看
3. **改进技能** — 根据反馈重写，重点：
   - 从反馈中**泛化**，不要只修特定例子
   - 保持 prompt 精简，删掉不产生价值的部分
   - **解释为什么**，不要只说"必须"
   - 检查是否所有测试用例都在重复写同样的辅助脚本——那是技能该bundled的信号

### 量化基准

如果做了测试，记录：
- 通过率
- 耗时
- token 消耗

## 改进技能

### 核心改进原则

1. **泛化反馈**：创建的是会被使用成千上万次的技能，不要只对少数例子过拟合
2. **保持精简**：删除不产生价值的部分
3. **解释为什么**：今天的 LLM 很聪明，给好的 harness 能超预期发挥
4. **bundled 脚本**：如果所有测试用例都独立写了同样的辅助脚本，那是技能该bundled的信号

### 迭代循环

1. 改进技能
2. 重新运行所有测试（包括 baseline）
3. 用 `--previous-workspace` 启动 reviewer
4. 等用户评审
5. 读反馈，再改进，重复

直到：用户满意 / 所有反馈为空 / 没有实质性进展

## 描述优化（可选）

技能完成后可以优化描述字段，提高触发准确率。

触发机制：Claude 根据 description 决定是否consult技能。太简单的一次性任务可能根本不触发——触发测试的query要足够 substantive。

**原则：** 描述要短，匹配质量高，不要浪费 turn-zero 预算。
