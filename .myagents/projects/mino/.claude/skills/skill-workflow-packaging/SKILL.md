# Skill Workflow Packaging

## 核心原则

技能是一个**受控的工作流包**，不只是 markdown 文档。
- 必须可发现
- 必须可触发
- 必须有触发条件
- 必须在 transcript 中清晰呈现

## 好技能的构成要素

1. **name** — 简洁、一听就懂、无歧义
2. **description** — 高信号触发语言，不是营销文案
3. **whenToUse** — 明确什么时候触发，不是模糊的使用建议
4. **触发词** — 列出精确的触发短语

## 触发语言原则

❌ 模糊：
- "当需要处理文档时使用"
- "用于各种文档操作任务"

✅ 高信号：
- "用户提到 Word doc、.docx、或要求生成报告"
- "提到 PDF 提取、PDF 解析、或 PDF 转 Markdown"

## 执行模式

- **Inline**：简单任务、token 预算充足
- **独立执行**：复杂工作流、需要隔离、防止污染主推理路径

## 触发条件优化

触发描述要包含：
1. 明确的动作词（搜索、创建、编辑、审核...）
2. 具体的输入形式（GitHub 链接、PDF 文件、URL...）
3. 具体的输出形式（报告、文档、代码...）

## 常见失败模式

- 触发描述太模糊，永远不会可靠触发
- 技能描述太长，消耗 turn-zero 预算
- 提到技能但不调用 Skill 工具
- 独立执行的技能没有进度报告，像黑箱

## 用于审核现有技能

检查清单：
1. 触发描述是否足够精确？
2. 是否与其他技能重叠？
3. 执行模式是否合理？
4. 是否需要 references/ 子目录？
